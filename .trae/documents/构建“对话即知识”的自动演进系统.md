# 构建“对话即知识”的自动演进系统 (Auto-Evolving Knowledge System)

为了实现你构想的“以用户输入为源，自动构建巨大知识网络”的系统，我们需要打破目前“文档入库”和“对话问答”的界限，建立一套**从交互中实时汲取知识**的机制。

这确实是一个漫长的积累过程，我们将通过以下架构变革来支撑这一愿景：

## 核心理念

**"Every Interaction is a Knowledge Ingestion" (每一次交互都是一次知识注入)**
用户的输入不再仅仅是稍纵即逝的查询指令，而是被视为需要被解析、提取并永久编织进知识图谱的原始素材。甚至应该是整个是一个toc的产品，为单个人服务。用户的输入只是最符合用户偏好的内容也可以被视为真值，其他后面可以自动的去检索和分析相关信息做一些理解和探索。为人类提供第二个“大脑”或者说私人助理。

## 技术实现方案

### 1. 新增核心模块：`MemoryPipeline` (记忆管道)

我们将创建一个异步处理管道，专门负责“消化”用户的对话。

* **文件位置**: `raganything/memory_pipeline.py`

* **功能**:

  * **Buffer & Filter**: 缓存用户输入，过滤掉无意义的寒暄（如"你好"、"谢谢"）和纯指令（如"把字变大"）。

  * **Fact Extraction**: 利用 LLM 从对话中提炼**陈述性事实（Facts）和**用户偏好（Preferences）还有其他相关有用信息。

  * **Graph Injection**: 调用 `HyperGraphRAG` 的增量更新接口，将提炼出的知识作为新节点插入图谱。

### 2. 架构流程改造 (The Loop)

在 `api_multi_document.py` 中建立闭环：

1. **User Input**: 用户发送消息。
2. **Immediate Response**: 系统正常检索并回答（保证响应速度）。
3. **Async Learning (后台学习)**:

   * 启动 `BackgroundTask`。

   * 分析刚才的 Input 和 Output。

   * **自动链接 (Auto-Linking)**: 如果用户提到了 "Project A"，而图谱里已有 "Project A" 的文档，系统会自动将这段对话挂载到该文档节点上。

   * **知识生长**: 如果用户提供了新信息（如“Project A 的截止日期改到了下周”），系统会更新图谱中该实体的属性。

### 3. 长期记忆策略 (Consolidation)

为了模拟“漫长的构建过程”，我们将引入**记忆整理**机制：

* **短期记忆 (Working Memory)**: 当前会话的上下文，保存在 Session Graph 中。

* **长期记忆 (Long-term Memory)**: 定期（或在会话结束时）将高价值的 Session 知识**合并 (Merge)** 到 Global Graph 中，使其成为永久知识库的一部分。

## 执行计划 (Roadmap)

### 第一阶段：基础设施搭建 (本次执行)

1. **创建** **`MemoryConsolidator`** **类**: 实现从对话文本中提取实体和三元组的逻辑。
2. **集成 FastAPI 后台任务**: 在 `/query` 接口后挂载学习任务，确保不阻塞用户聊天。
3. **设计提取 Prompt**: 编写专门用于从对话流中提取知识的 Prompt。

### 第二阶段：UI 感知与反馈 (未来规划)

1. **学习状态展示**: 在 UI 上显示“系统正在消化新知识...”。
2. **知识确认**: 对于重要的知识更新（如修改事实），主动向用户确认（“我记住了，Project A 的截止日期是下周，对吗？”）。

这个架构将彻底改变系统的性质：它不再是一个静态的“查书工具”，而是一个**随着你的使用而不断成长、越来越懂你的“第二大脑”**。
